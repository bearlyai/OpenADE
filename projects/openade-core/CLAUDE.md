# OpenADE Core

Standalone Go product kernel for OpenADE.

- This package is the destination backend described in ../../goal.md.
- Run `make check` from this directory before shipping Core changes; it runs the self-tested non-test `any`/`interface{}` scanner and `go test ./...`.
- Keep the wire shape compatible with projects/runtime-protocol unless that contract is intentionally changed.
- Use `Sec-WebSocket-Protocol: bearer.<token>` for WebSocket bearer tokens; do not put bearer tokens in query strings. `core.HTTPServer` must echo the selected bearer subprotocol during WebSocket accept after authorization, otherwise standards-compliant browser/Node clients reject the handshake even when auth succeeds.
- Keep logs sanitized. Do not log prompts, file contents, env values, bearer tokens, private paths where avoidable, or task payloads.
- Slow-operation telemetry must separate queue wait from handler time and include sanitized service, method, request id, connection id, failure state, and error code fields. Do not log prompts, content, tokens, private paths, or raw unbounded client-controlled ids.
- Do not use direct `any` or `interface{}` in non-test Go code. Use concrete DTOs, `json.RawMessage` for validated opaque JSON, or `core.JSONPayload` only at the runtime protocol boundary. Any unavoidable runtime/stdlib dynamic boundary must have a nearby real source comment using `openade-allow-explicit-any: concrete reason` and a matching reviewed entry in `scripts/check-no-explicit-any.mjs` so the scanner can enforce the rule and report the exception count.
- `OPENADE_CORE_PERMISSION_PROFILE=paired` applies the reviewed paired-client permission set unless `OPENADE_CORE_PERMISSIONS` supplies an explicit list. The paired profile grants high-level OpenADE product reads and normal task-thread actions while denying raw Yjs, repo admin, file writes, git commit, terminal, import/migration, snapshot-create, action-event plumbing, HyperPlan plumbing, process start/stop, raw runtime notifications, process output notifications, and PTY output notifications. Paired clients should consume product-level `openade/*` notifications instead of raw `runtime/*`, `process/*`, or `pty/*` streams. Treat this as a safe attach profile, not the final per-device/session role model.
- Device bearer tokens are SHA-256 hashed in SQLite `devices.token_hash`. `core.HTTPServer` must authenticate the configured trusted token first, then stored non-revoked device tokens through `product.ConfigureDeviceAuthentication()`, which sets paired permissions and the connection device id. `remote/pairing/start` is trusted-only and creates a short-lived one-use pairing session plus companion-compatible payload; unauthenticated `POST /v1/pair` can only exchange that session token for a stored device token through `Service.ConfigurePairing()`. `remote/device/list`, `remote/device/revoke`, and `remote/device/dropAll` are trusted/admin methods; paired devices only get `remote/device/selfRevoke`. Revoking a device or dropping all devices must close active runtime WebSocket connections for those device ids; self-revoke must return its success response before closing the caller socket.
- Product state belongs in SQLite/blob storage once storage lands; do not recreate hot Yjs full-document reads in Go.
- `internal/storage` owns the SQLite migration/store layer. Extend its versioned migrations for product state, materialized previews, runtime records, sessions/devices, usage, and blob metadata instead of adding ad hoc database setup elsewhere.
- `internal/host` owns scoped host operations such as git, files, search, process, and PTY reads as they move into core. Resolve repo/task paths server-side from product records; never accept raw host paths from paired clients.
- `internal/product` owns the OpenADE product runtime methods backed by storage and scoped host adapters. Current methods include runtime list/read/reconcile/stop, pairing start/exchange, remote device admin/self-revoke, trusted MCP server settings read/replace/upsert/delete, trusted per-repo cron install-state read/replace, snapshot/project/task reads, task create, task device environments in task reads, task resource inventory/image/snapshot-patch reads, Core-owned `task_image` blob writes and trusted staged-preview reads, trusted legacy file/directory imports plus staged-image GC and resource import orchestration, snapshot event creation with Core-owned `snapshot_patch` blob writes and trusted legacy patch directory imports, project file tree/read/write/fuzzy/content search reads, project git info/branches/summary reads, task git summary/scopes reads, task changes/diff/file-pair reads, task git log/commit-files/file-at-treeish/commit-file-patch reads, task git commit, initial task environment setup/prepare, project process config/list/start/reconnect/stop, task terminal start/reconnect/write/resize/stop, queued turns in task reads, queued-turn enqueue/reorder/cancel plus trusted non-executing legacy import and executor-backed drain, initial turn start/interrupt and review start, task title generation, trusted usage preview recalculation/backfill, action-event create/stream/append/complete/error/stopped/reconcile/execution-update primitives, HyperPlan sub-execution add/stream/update/reconcile-label primitives, repo create/update/delete, comment create/edit/delete, task delete, and the narrow `openade/task/metadata/update` subset for title/closed/viewed/event/usage preview metadata; extend these contract-first rather than adding shell-specific shortcuts.
- Project file/search methods resolve omitted `taskId` to the repo path, head tasks to the repo path, and prepared worktree tasks to the latest completed stored device environment worktree. Unprepared worktree tasks fail with `invalid_params` instead of falling back to the repo root.
- Project process config reads resolve omitted `taskId` to the repo path, head tasks to the repo path, and prepared worktree tasks to the latest completed stored device environment worktree. Unprepared worktree tasks fail with `invalid_params` instead of falling back to the repo root.
- Task git summary/scopes, changes/diff/file-pair, git log/commit-files/file-at-treeish/commit-file-patch, and git commit resolve head tasks to the repo path and worktree tasks to the latest completed stored device environment worktree. Unprepared worktree tasks fail with `invalid_params` instead of falling back to the repo root. Task git scopes include `branch:*` entries selected through the log `ref` parameter and `worktree:*` entries resolved server-side from `git worktree list --porcelain`; never accept a raw worktree path from the client.
- `openade/task/environment/prepare` owns the initial core worktree lifecycle slice for headless runtime environments. It creates/reuses `openade/<task-slug>` git worktrees under the configured core worktree base directory, persists `task_device_environments`, appends setup events for worktree tasks, and honors `clientRequestId` idempotency. Keep tests on real temp git repos; do not mock worktree behavior.
- `openade/task/git/commit` is a trusted/local product mutation, stages all scoped task workdir changes, honors `clientRequestId` idempotency, returns committed/nothing-to-commit/failed status, and must not be granted to paired devices until scoped commit permissions are explicit.
- `openade/project/file/write` is a trusted/local product mutation, resolves head/prepared-worktree task scopes through stored device environments, honors `clientRequestId` idempotency, supports utf8/base64 content plus optional parent-directory creation, and must not be granted to paired devices until the permission model explicitly allows scoped writes.
- `openade/project/process/list` returns parsed `openade.toml` configs, process definitions, cwd/config errors, and in-memory process instances across repo/head/prepared-worktree task scopes. `openade/project/process/start`, `openade/project/process/reconnect`, and `openade/project/process/stop` run only server-resolved definitions, buffer bounded stdout/stderr chunks, persist process runtime records, persist Unix process group ids when available, persist private Core-owned stdout/stderr capture file paths plus byte offsets, persist bounded stdout/stderr history in SQLite `runtime_output_chunks`, emit `process/started`, `process/output`, `process/exit`, `process/error`, `runtime/created`, `runtime/completed`, `runtime/failed`, and `runtime/stopped` notifications, enforce repo/task process scope, and honor `clientRequestId` idempotency for start/stop. Unix project processes are started in their own process group and stop paths signal that group before falling back to the direct process, so repo-declared commands cannot leave child processes behind. `openade/project/process/reconnect` uses live memory first, then durable output for completed/orphaned matching process runtimes, but does not resurrect explicitly stopped processes. `runtime/list`, `runtime/read`, `runtime/reconcile`, and `runtime/stop` expose those records without exposing private capture paths. Startup marks previously active persisted process runtimes as `orphaned`, adopts still-running Core-owned project processes when private capture files and offsets exist, tails those files into SQLite/runtime notifications, wires adopted processes into product/runtime stop through stored PID/PGID, and marks adopted processes `completed` when the PID exits without a recovered exit code. Startup and `runtime/reconcile` still mark orphaned/running process runtimes with verified-dead PIDs as `stopped`.
- `openade/task/terminal/start`, `openade/task/terminal/reconnect`, `openade/task/terminal/write`, `openade/task/terminal/resize`, and `openade/task/terminal/stop` run a real PTY scoped to the server-resolved task workdir, use the deterministic `openade-task-terminal-*` id format, persist `pty:*` runtime records, persist bounded PTY output history in SQLite `runtime_output_chunks`, emit `pty/started`, `pty/output`, `pty/exit`, `pty/killed`, and runtime notifications, and honor `clientRequestId` idempotency for mutations. Terminal reconnect uses live memory first, then durable output for completed/orphaned matching PTY runtimes. Startup and `runtime/reconcile` mark verified-dead PID-backed PTY runtimes as stopped with `terminal process is no longer running`, while keeping their durable output reconnectable; explicitly user-stopped terminals still return not found. Task terminals are trusted/local product methods and must not be granted to paired devices until the permission model explicitly allows interactive shell access.
- Action-event persistence methods write SQLite task-event rows and preserve the existing action payload JSON shape for TypeScript/UI compatibility. `openade/action/create`, `openade/action/stream/append`, `openade/action/complete`, `openade/action/error`, `openade/action/stopped`, `openade/action/reconcileRuntime`, and `openade/action/execution/update` are server-side execution-state primitives. Executor-driven turns use the same helpers to append stream events, update session/git refs, and settle terminal action status.
- HyperPlan sub-execution persistence methods mutate the existing action payload JSON shape under `hyperplanSubExecutions`. `openade/hyperplan/subExecution/add` validates primitive/status and required harness/model/step fields, duplicate step ids are no-ops, stream append dedupes by stream event id, update only applies provided typed string fields, and reconcile-label updates ignore missing steps without rewriting the event.
- `openade/turn/start` owns the initial core turn-start boundary: it creates/reuses tasks, creates an in-progress action event, persists a running `agent` runtime record, emits task/runtime/working-task notifications, and can run a configured `AgentExecutor` asynchronously. Executor output is persisted to the action event stream, executor session/git refs are persisted under `execution`, completed/failed/stopped results settle both the action event and runtime record, and `runtime/stop` plus product-level `openade/turn/interrupt` cancel the active executor after persisting terminal task/runtime state. Completed executor turns drain the next queued turn by atomically claiming the oldest queued row, creating the queued action/runtime, and running it through the same executor path; running queued turns settle to completed/error/stopped with their action runtime. `openade/review/start` builds the classic review prompts from Core task events/snapshots, starts the review in read-only mode, persists the review action/runtime, and auto-starts a read-only ask follow-up from the review output. The `openade-core` binary can enable the process-bound `CommandAgentExecutor` with `OPENADE_CORE_AGENT_WORKER_COMMAND`, preferably a JSON array such as `["node","/path/to/projects/harness/dist/worker.js"]` after building `@openade/harness`; Core sends one JSON `start` envelope on stdin and expects NDJSON `stream`, `execution`, and `result` messages on stdout. The command executor persists worker `pid`, Unix `pgid`, `processStartedAt`, and a private recovery transcript path onto the agent runtime record through `runtime/updated`; startup and `runtime/reconcile` use the persisted Unix `pid` to stop and settle stale agent runtimes when the worker process is verified dead. The TypeScript worker lives in `projects/harness/src/agent-worker.ts`, runs Claude Code/Codex through the shared harness interface, honors Core's `readOnly` flag as `HarnessQuery.mode = "read-only"`, emits the same persisted execution-event shapes the classic renderer already consumes, and appends the same worker-protocol NDJSON to `OPENADE_AGENT_WORKER_RECOVERY_FILE` when Core provides it. Startup marks previously active agent runtimes `orphaned`, replays any command-worker recovery transcript that contains a terminal `result`, reactivates still-live worker processes with non-terminal transcripts, tails their recovery files until a terminal result arrives, and wires adopted workers into `runtime/stop`/`openade/turn/interrupt` through stored PID/process-group termination. Verified-dead worker runtimes are settled to `stopped`; live orphaned Unix worker process groups are only terminated as fail-closed recovery when there is no resumable recovery transcript. When `enabledMcpServerIds` are present, Core resolves matching enabled rows from the SQLite `mcp_servers` setting into `mcpServerConfigs` for the worker; HTTP rows merge `oauthTokens.accessToken` into an Authorization header and disabled/unselected rows are omitted. Turn images remain lightweight refs in persisted action/queued-turn payloads, but Core expands task-owned `task_image` blob metadata into base64 prompt image sources for the worker after proving the image is referenced by the task. Electron can package and opt-in launch the Core binary for managed-Core testing, but Core still needs broader migration UX around `openade/import/legacyResources`, packaged restart-during-live-turn smoke coverage, and full rollout cleanup before execution migration is complete.
- `openade/settings/mcpServers/read`, `openade/settings/mcpServers/replace`, `openade/settings/mcpServers/upsert`, and `openade/settings/mcpServers/delete` own Core MCP server rows in SQLite settings. They preserve the desktop row shape needed by settings UI, including preset ids, health state, timestamps, HTTP headers, OAuth tokens, stdio args/env/cwd, and enabled state, while validating IDs, transport-specific fields, URLs, and timestamps. These methods are trusted/local because they expose secrets and local process commands; paired devices must not receive them unless a future settings-admin role is explicitly designed.
- `openade/settings/personal/read` and `openade/settings/personal/replace` own Core personal settings in SQLite settings for clean managed-Core installs. They preserve the classic desktop personal-settings shape, including env vars, theme, telemetry, onboarding, dev toggles, new-task defaults, pinned task ids, and HyperPlan preferences. These methods are trusted/local because they expose env values and device preferences; keep them out of paired-device grants until an explicit settings-admin role exists. Core applies those env vars directly to Core-owned project processes, task terminals, task title generation, turn/review command-agent workers, and queued/review follow-up workers; clean-Core renderer startup must not push them back through Electron `host/subprocess/setGlobalEnv`.
- `openade/cron/installState/read` and `openade/cron/installState/replace` own per-repo cron installation state in Core SQLite settings so renderer cron UI can move off raw `data/cron/*.json` files. They validate repo ids through Core storage, preserve existing path/name-shaped cron ids, validate timestamps, honor `clientRequestId` idempotency for replace, and must stay trusted/local until Core has an explicit scheduler permission model. The Core daemon starts the headless cron scheduler with `Service.StartCronScheduler()`: it scans SQLite repos and installed cron state, resolves matching cron definitions from server-side `openade.toml` parsing, computes due five-field schedules with `robfig/cron`, updates `lastRunAt` before starting a turn, uses deterministic per-occurrence `clientRequestId`s, and starts work through the normal `openade/turn/start` service path. Keep renderer cron scheduling disabled for clean managed-Core sessions and do not add a paired/browser runtime method that can drive the scheduler directly.
- Product mutations must honor `clientRequestId` idempotency. Repeated successful mutation requests with the same method scope and client request id should return the retained result instead of running the mutation again; failed attempts should not be retained.
- `openade/task/create` creates SQLite task and preview rows plus the initial device environment/setup event transactionally, preserves existing task data on retry instead of overwriting collisions, emits task/snapshot notifications only for new rows, and uses the same deterministic `repoId + clientRequestId` task id format as the TypeScript host path.
- `openade/task/read` comment DTOs must preserve Core's existing `body`/`anchor` fields and also expose legacy-compatible top-level `content`, `source`, `selectedText`, and `author` fields when `anchor_json` contains the comment anchor. This keeps imported SQLite-backed comments usable by old Yjs projection consumers while desktop callers migrate onto concrete comment DTOs.
- `openade/task/read` must project persisted task metadata JSON for valid `createdBy`, `enabledMcpServerIds`, `sessionIds`, and `cancelledPlanEventId` fields. Decode this metadata through concrete DTOs with tolerant readers and trim/filter empty legacy strings instead of exposing malformed records or adding loose dynamic typing. `openade/task/metadata/update` may merge `sessionIds` and `cancelledPlanEventId` into that metadata for legacy import/parity; keep `queuedTurns` on first-class queue rows and keep `enabledMcpServerIds` create-time until a broader settings/update contract is designed.
- `openade/task/title/generate` resolves repo/task/workdir through Core storage, builds the concise title prompt with bounded recent event context, runs the configured executor in read-only title mode when possible, falls back to the existing title or truncated description, persists through Core task metadata storage, emits task/preview notifications, and honors `clientRequestId` idempotency.
- `openade/task/usage/backfill` is the trusted/local bulk task usage materialization path for clean managed-Core stats backfill. It accepts optional repo/task filters, skips previews that already have usage v2 plus duration unless `force` is set, scans SQLite task-event payloads, computes the `OpenADETaskPreviewUsage` v2 preview shape from action and HyperPlan execution streams, persists `task_previews.usage_json`, emits task/preview notifications, and must stay out of paired grants until usage recalculation permissions are explicit. `openade/task/usage/recalculate` remains the single-task trusted/local primitive; UI backfill should prefer the bulk method. Prefer explicit `costUsd` values from execution events; do not reintroduce renderer full-history scans for clean Core.
- `openade/task/resourceInventory/read` is backed by SQLite task/event rows, active Core runtime records, and optional git inspection for worktree merge metadata. `isRunning` must be true when any `starting` or `running` runtime has `scope.ownerType = "openade-task"` and `scope.ownerId` equal to the task id, and false for terminal/orphaned or other-task runtimes.
- `openade/task/image/write` is a trusted/local product mutation for pre-task image staging. It accepts sanitized image ids/exts, matching image media types, and base64 bytes; writes under the configured Core blob directory as `task_image` metadata; treats same-id/same-content retries as idempotent; and rejects same-id/different-content conflicts. Paired devices must not receive this grant until upload permissions are explicit.
- `openade/task/image/staged/read` is a trusted/local product read for pending composer previews. It validates sanitized image ids/exts, reads only matching Core-owned `task_image` blob metadata, returns base64 bytes for unreferenced staged images, returns `data: null` for missing/mismatched blobs, and must stay out of paired-device grants until upload/read permissions are explicit.
- `openade/task/image/importLegacy` is a trusted/local migration primitive. It accepts a sanitized target image id/ext/media type plus a local `sourcePath`, copies the source file into the Core blob directory as `task_image` metadata, preserves the source file, reuses write idempotency/conflict semantics, and must not be granted to paired devices because it accepts a raw host path.
- `openade/task/images/importLegacy` is the trusted/local bulk production image import primitive. It scans Core task events and queued turns for referenced image ids, dedupes refs, copies matching `imageId.ext` files from a local legacy image directory into Core-owned `task_image` blobs, reports imported/already-imported/missing/conflicted/failed counts, preserves source files, and must not be granted to paired devices because it accepts a raw host directory.
- `openade/task/images/gcStaged` is trusted/local staged-image garbage collection. It scans Core-owned `task_image` blob metadata, retains any image id referenced by task events or queued turns, retains young unreferenced images by default, supports dry-run reporting, deletes metadata only when the scanned blob row is unchanged, and removes orphaned files best-effort. Keep it out of paired-device grants because it is a maintenance mutation over local blob storage.
- `openade/task/image/read` must verify the image id/ext is referenced by the task's events or queued turns before reading a `task_image` blob path from storage.
- `openade/snapshot/create` is a trusted/local product mutation. It must prove the referenced action event belongs to the task, externalize non-empty `fullPatch` content into a Core-owned `snapshot_patch` blob under the configured blob directory, store only lightweight `patchFileId` references in task events, emit task/snapshot notifications, and honor `clientRequestId` idempotency.
- `openade/task/snapshot/patch/read`, `openade/task/snapshot/index/read`, and `openade/task/snapshot/patch/readSlice` must resolve a snapshot event from SQLite task events before reading inline `fullPatch` content or a `snapshot_patch` blob path from storage.
- `openade/task/snapshots/importLegacy` is the trusted/local bulk production snapshot patch import primitive. It scans Core task events for valid `patchFileId` references, dedupes refs, copies matching `patchFileId.patch` files from a local legacy snapshot directory into Core-owned `snapshot_patch` blobs, reports imported/already-imported/missing/conflicted/failed counts, preserves source files, and must not be granted to paired devices because it accepts a raw host directory.
- `openade/task/sessions/importLegacy` is the trusted/local bulk production harness transcript import primitive. It scans Core task events and metadata for task-owned Claude Code/Codex session ids, locates real Claude project JSONL files under `CLAUDE_CONFIG_DIR`/`~/.claude` or explicit `claudeConfigDir` and Codex session JSONL files under `CODEX_HOME`/`~/.codex` or explicit `codexHome`, copies matching files into Core-owned `harness_session` blobs under the configured blob directory, reports imported/already-imported/missing/conflicted/failed counts, preserves source files, and must not be granted to paired devices because it accepts raw host directories and imports raw transcripts.
- `openade/import/legacyResources` is the trusted/local resource migration orchestrator. It accepts a local legacy `dataDir` or explicit image/snapshot/session directories, defaults data-root image/snapshot paths to `dataDir/images` and `dataDir/snapshots`, can run session import when `importSessions` or explicit session roots are provided, reuses the bulk image, snapshot, and harness-session import primitives, skips missing derived subdirectories, reports typed skipped entries, and must not be granted to paired devices because it accepts raw host directories.
- `openade/task/delete` deletes SQLite task state and supports `deleteSnapshots`/`deleteImages` for Core-owned `snapshot_patch` and `task_image` blobs, `deleteSessions` for best-effort Claude Code/Codex CLI transcript cleanup, and `deleteWorktrees` for Core-managed or Git-registered imported task worktrees. Session cleanup derives task-owned session ids from the same task resource inventory fields as the TypeScript runtime: action execution sessions, HyperPlan sub-execution sessions, and task metadata `sessionIds`. It deletes Claude Code files under `CLAUDE_CONFIG_DIR`/`~/.claude` and Codex files under `CODEX_HOME`/`~/.codex`, ignores unknown harness ids, and must not touch Core device/client sessions or imported `harness_session` blobs. Worktree cleanup validates the deterministic Core worktree-base slug path or requires `git worktree list --porcelain` to prove the imported path is registered on the exact `openade/<slug>` branch before task deletion. After SQLite deletion, Core removes the worktree and exact branch best-effort. Unsafe external paths, detached worktrees, branch mismatches, unregistered paths, and repository-root paths are rejected instead of removed.
- Queued turns are stored as first-class rows. Do not write `queuedTurns` back through task metadata; enqueue, reorder, cancel, claim, running-event assignment, and terminal status updates go through explicit queue storage methods. Queue drain must preserve serialized execution: only one active `agent` runtime per task should claim queued work.
- `openade/queued-turn/importLegacy` is a trusted/local migration primitive. It preserves legacy queued-turn rows and payload fields through `queued_turns` without calling queue drain or starting an executor. Keep it out of paired-device grants; normal user queue writes should keep using enqueue/reorder/cancel.
- Electron should become a launcher/native-shell adapter for this core, not a second product backend.
- Tests should use real HTTP/WebSocket transports and temp directories. Mock-only tests do not count as migration confidence.
- `TestProductReadPerformanceBudgetsOverRuntime` and `TestProductHostOperationPerformanceBudgetsOverRuntime` are hard budget gates for real runtime reads. Keep them on the WebSocket path with production-shaped SQLite/temp-git/filesystem data; do not replace them with direct service calls or mocks when tuning task-open, git summary, process list, or fuzzy-search performance.
