# Electron Desktop Shell Guidance

## Preload Contract

- `src/preload-api.ts` owns the browser-safe `OpenADEAPI` shape exposed by `src/preload.ts` and consumed by `projects/web/src/vite-env.d.ts`.
- When changing the contextBridge surface, update `src/preload-api.ts` first and keep `src/preload.ts` using `satisfies OpenADEAPI` so the renderer global cannot drift from the actual preload object.
- `openadeAPI.core.runtimeEndpoint` is the narrow rollout bridge for attaching the renderer's OpenADE product client to a Go Core WebSocket endpoint, and `openadeAPI.core.rolloutState` exposes only sanitized rollout status/reason/source booleans for settings and telemetry. A valid `OPENADE_CORE_RUNTIME_URL` exposes an externally managed Core endpoint unless `OPENADE_DISABLE_OPENADE_CORE=1` is set. Managed Core launch lives in `src/modules/runtimeCore.ts`: `OPENADE_CORE_MANAGED=1` or `OPENADE_USE_OPENADE_CORE=1` explicitly starts Core when no external endpoint is configured; clean production installs also auto-start the packaged Core binary when the legacy Yjs storage directories have no documents. Existing Yjs-backed installs stay on the trusted IPC path while legacy Yjs documents exist unless a clean Core data/parity import and clean legacy resource import write the narrow accepted-import marker at `~/.openade/data/core/legacy-yjs-import-accepted.json`. The accepted marker reports `legacy-yjs-migration-accepted`; unaccepted legacy docs must report `legacy-yjs-documents` rather than leaking paths. The marker writer is the trusted-local `host/core/legacyYjsMigration/accept` runtime method and must stay out of paired-device grants. Managed Core prefers the built `dist/openade-core/openade-core` binary in dev and the packaged `dist/openade-core/openade-core` binary under `process.resourcesPath` in packaged apps, falls back to `go run ../openade-core/cmd/openade-core` only for explicit managed launches when no built binary is available, defaults to `127.0.0.1:37376`, and publishes `OPENADE_CORE_RUNTIME_URL`/`OPENADE_CORE_TOKEN` before preload reads them. When no explicit `OPENADE_CORE_AGENT_WORKER_COMMAND` is configured, managed Core should also receive the packaged `dist/harness-worker/worker.js` command plus `ELECTRON_RUN_AS_NODE=1` so Core-owned turns can spawn the typed harness worker without a user Node install. `main.ts` must call `loadRuntimeCore({ isDev })` before `loadExecutorWindow()` because the preload selects the product runtime synchronously during BrowserWindow startup. Raw desktop host APIs still use the trusted local IPC runtime until Core has full host parity.

## Code Host Bridge Contracts

- `src/modules/code/hostBridgeTypes.ts` owns browser-safe DTOs for desktop-specific trusted host utilities such as binaries, platform, shell, frame colors, and code-module capability probes.
- `src/modules/code/gitBridgeTypes.ts` owns browser-safe DTOs for raw trusted-local `git/*` bridge methods. Product-scoped task git scope/read/commit DTOs still come from `projects/openade-module/src/types.ts`, and scoped git scope reads must not expose raw worktree paths to renderers or paired devices.
- `src/modules/code/yjsStorage.ts` is the desktop Yjs persistence boundary. It intentionally coalesces in-flight document loads and keeps a short invalidated-on-write read cache long enough to absorb sub-15s renderer refresh loops, so runtime product reads do not repeatedly load multi-megabyte task/repo documents during one task switch or idle notification burst. Do not remove that cache or reintroduce per-collection task document loads without a real performance regression test.
- `src/modules/code/git.ts` intentionally coalesces identical `getRuntimeGitSummary()` and `isRuntimeGitDirectory()` calls and keeps only short fresh result windows, invalidated by git/worktree mutations, so task route/tray refreshes do not launch duplicate expensive git probes for the same repo. `isRuntimeGitDirectory()` also shares a short `gh auth status` cache; do not put a GitHub CLI auth probe back inside every repo git-info read.
- Electron slow-operation logs for runtime requests, Yjs document I/O, repeated Yjs loads, and git summaries must stay sanitized. Runtime slow request logs must preserve the RuntimeServer observer fields, including service, method, sanitized request id, total duration, queue wait, handler time, connection id, failed state, and sanitized error code. Document ids, byte counts, and durations are acceptable for Yjs logs; repo paths, prompt text, file contents, tokens, and env values are not.
- `src/modules/deviceConfigTypes.ts` owns browser-safe device config result DTOs. Snapshot patch index DTOs and indexing/slicing helpers live in `projects/openade-module/src/snapshotPatchIndex.ts`; `src/modules/code/snapshotsIndex.ts` should stay a compatibility re-export for Electron callers.
- `openade.toml` process/cron DTOs and parser/serializer live in `projects/openade-module/src/types.ts` and `projects/openade-module/src/procs.ts`. Electron procs modules should alias or re-export those product-owned contracts, not keep a second parser or type set.
- Scoped OpenADE project file tree/read/write/filename-search/content-search behavior, optional task scoping, task workdir resolution, path containment, hidden/generated filtering, file-size limits, and file metadata live in `projects/openade-module/src/scopedProjectHost.ts`. The companion runtime gateway should import those product-owned helpers instead of carrying Electron-only copies.
- Scoped OpenADE project process definition building, cwd containment, registration/scope matching, timeout policy, and runtime process DTO normalization live in `projects/openade-module/src/scopedProjectProcesses.ts`. Electron may keep its process start/reconnect/kill plumbing, but should import the shared helpers for parsed `openade.toml` configs and product process result shapes. Companion `openade/project/process/list` intentionally keeps short read caches for parsed `openade.toml` discovery and raw runtime process lists; invalidate those caches on process start/stop and procs file writes rather than reintroducing repeated repo/process scans during UI refresh bursts.
- Scoped OpenADE task terminal id derivation and raw PTY base64/plain-text conversion helpers live in `projects/openade-module/src/scopedTaskTerminal.ts`. Electron may keep PTY lifecycle plumbing, but should not maintain a second terminal hash or encoding implementation.
- Stable OpenADE task and queued-turn ids derived from `clientRequestId` live in `projects/openade-module/src/clientRequestIds.ts`. Electron should import those helpers instead of carrying request-id hashing logic.
- OpenADE task snapshot patch/index/slice read semantics live in `projects/openade-module/src/taskSnapshotPatchReads.ts`. Electron should keep storage plumbing in `src/modules/code/snapshots.ts` and pass those loaders to the product helper instead of duplicating inline/external patch read rules.
- OpenADE task resource inventory extraction lives in `projects/openade-module/src/taskResourceInventory.ts`. Electron may compute host-specific worktree branch merge status, but should not duplicate snapshot/image/session event scans in `runtimeGateway.ts`.
- Trusted MCP settings rows are product settings. `runtimeGateway.ts` may expose the legacy `code:mcp_servers` document through `openade/settings/mcpServers/read|replace|upsert|delete` by using the shared OpenADE module Yjs mutation helpers, but renderer settings UI should call the product settings methods rather than raw Yjs documents.
- Trusted personal settings are also product settings. `runtimeGateway.ts` may expose legacy `code:personal_settings` through `openade/settings/personal/read|replace` for compatibility, but clean managed Core should persist them in Core SQLite and the renderer should not open the legacy personal-settings Yjs document during clean-Core startup. Clean managed Core also owns applying personal-settings env vars to Core-owned subprocesses; do not reintroduce `host/subprocess/setGlobalEnv` pushes on that path.
- Low-level harness IPC event/query DTOs and harness install status DTOs live in `projects/harness/src/types.ts` as `HarnessIpc*` and `HarnessInstallStatus`. Electron harness bridge code should import those types from `@openade/harness` instead of declaring renderer-matching copies.
- Renderer wrappers under `projects/web/src/electronAPI` should alias those types instead of re-declaring matching Electron main-process interfaces.

## Packaged Smoke Tests

- `tests/smoke.spec.ts` drives the packaged app artifact, not the dev server. Rebuild with `npm run build`, `npm run build:web`, and `NONOTARY=1 CSC_IDENTITY_AUTO_DISCOVERY=false npx electron-builder --mac --dir` before trusting smoke results after main/preload/web changes.
- `npm run build` now builds `projects/openade-core/cmd/openade-core` into `dist/openade-core/openade-core`, copies `projects/harness/dist` to `dist/harness-worker`, and then bundles Electron main/preload. electron-builder includes both `dist/openade-core/**` and `dist/harness-worker/**` as extra resources. Use `OPENADE_SKIP_CORE_BUILD=1` only for narrow JS build/debug work where the packaged Core binary is irrelevant.
- Packaged smoke is expected to cover bundled UI boot, runtime IPC initialization, repo creation, task workflow/reload, scoped file operations, and scoped project process list/start/reconnect-output/stop through the built Electron host.
- Packaged smoke also includes a default-on clean-install managed-Core path: it launches Electron with isolated temp Core/Yjs/user data and no explicit `OPENADE_USE_OPENADE_CORE` or `OPENADE_CORE_MANAGED` opt-in so the auto-managed policy must choose Core. Targeted smoke variants may still set those flags when explicit opt-in behavior is under test. The clean-install smoke must assert the sanitized rollout state reports `connected`/`managed`/`managed-core` with `automatic: true`, wait for renderer `app_opened` telemetry proving the runtime product store is enabled, ready, and has a snapshot, then connect from the renderer to `openadeAPI.core.runtimeEndpoint` over a real browser WebSocket. That smoke must keep exercising real packaged Core product paths against isolated temp Core data and a real temp git repo: `initialize`, snapshot read, repo create, task create/read, `openade/turn/start` through the packaged harness worker plus `runtime/read` proof of the completed `openade-turn:<eventId>` agent runtime, comment create, task metadata update, scoped file tree/read/write, fuzzy/content search, project git info/branches/summary, task git summary/changes/diff/file-pair/log/commit, process config list/start/reconnect-output/stop, non-Windows terminal start/write/reconnect-output/resize/stop, and the runtime-product rollout telemetry reviewer. It must also navigate the packaged renderer to the Core-created task URL, prove the classic desktop task surface renders Core-created turn text while compact shared/mobile task surfaces remain absent, start a delayed Core-owned turn through the packaged worker, close/relaunch the app while that turn is live, prove Core recovers the turn to `completed` from the worker transcript after relaunch, and prove the same persisted Core task renders both the original and recovered turn text through the classic desktop route again. Keep this smoke separate from the legacy IPC workflow smoke until the classic desktop UI is fully Core-backed by default.
- Packaged desktop smoke must keep proving the classic desktop UI path. The migration target is classic desktop look/functionality over runtime APIs, not rendering the compact remote/mobile shell in Electron.
- Smoke tests may set `OPENADE_SMOKE_TEST=1` and `OPENADE_DISABLE_ACTIVE_WORK_UNLOAD_BLOCKER=1` so active-work quit/beforeunload prompts do not block Playwright teardown. Do not disable those blockers in normal app launches.
- Packaged smoke must set `OPENADE_YJS_STORAGE_DIR` to its temp user-data directory. `HOME`/`USERPROFILE` isolation alone is not enough to prove the packaged app is avoiding the developer's normal `~/.openade/data/yjs` documents.
- The smoke harness may use `OPENADE_SMOKE_DETERMINISTIC_HARNESS=1`; keep it guarded by `OPENADE_SMOKE_TEST=1` and limit it to deterministic packaged workflow coverage. `OPENADE_SMOKE_DETERMINISTIC_HARNESS_DELAY_PROMPT` plus `OPENADE_SMOKE_DETERMINISTIC_HARNESS_DELAY_MS` may delay only a matching deterministic prompt so managed-Core smoke can restart during a live worker turn without slowing unrelated smoke turns.
- In smoke mode, the preload exposes `openadeAPI.app.smokeTest` so the renderer records real analytics `track()` calls to local storage. `tests/smoke.spec.ts` must run `projects/web`'s `review:runtime-product-rollout` command against that export, proving the packaged classic desktop route emits a ready default-on `app_opened` and no runtime product fallback/error telemetry.

## Linux Startup Stability

- `build/afterPack.js` wraps the packaged Linux executable so display-backend flags are present before Electron initializes. Read it before changing Linux packaging, launch flags, or AppImage startup behavior.
- `src/modules/linuxDisplayBackend.ts` is the runtime fallback for non-wrapped development launches. It is not early enough by itself for packaged AppImage startup.
- Wayland sessions default to `--ozone-platform=x11` for packaged Linux builds because native Wayland startup has produced AppImage SIGSEGV failures. Users can override this with `--ozone-platform=...` or `OPENADE_LINUX_OZONE_PLATFORM=x11|wayland|auto`.
- `src/modules/sentryConfig.ts` keeps Sentry JavaScript reporting enabled while disabling Electron native minidump integrations on Linux. Do not re-enable Linux native crash reporting without validating AppImage startup on real GNOME/KDE Wayland sessions.
