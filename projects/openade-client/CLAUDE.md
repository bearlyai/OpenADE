# OpenADE Client

Target package for typed OpenADE project/task/turn client APIs.

- Should call OpenADE runtime methods through runtime-client.
- Dashboard and mobile should converge on this package.
- Do not duplicate companion-only domain request types here.
- Runtime transport clients own `initialize`; OpenADEClient assumes it is given a runtime-client-compatible transport and only owns typed snapshot/project/task reads, turn-start helpers, task create/update/delete product mutations, queued-turn enqueue/reorder/cancel plus trusted legacy-import mutations, action-event and snapshot-event persistence helpers, scoped project file tree/read/write/search/process helpers, scoped project git info/branches/summary reads, scoped task terminal helpers, scoped task git scope/read/commit helpers, scoped task environment prepare, scoped task title generation, scoped task image write/read plus trusted staged-image preview reads, single/bulk legacy-import, and staged-image GC helpers, trusted MCP server settings helpers, cron definition reads plus trusted cron install-state helpers, resource-inventory/snapshot patch read plus trusted legacy-import helpers, trusted harness-session and resource import orchestration helpers, and OpenADE-scoped notification filtering.
- `openade-contracts.json` is the package's current generated contract source for OpenADEClient method names, remote/session method names, product/remote notification names, runtime error codes, and request/response DTO mapping. Run `npm run generate:contracts` after changing OpenADEClient methods, remote device/session methods, notifications, error codes, or DTO mappings, and keep `npm run check:contracts` in typecheck so the generated `src/generated/openade-contracts.ts`, generated Core `internal/product/generated_contracts.go`, generated module `src/generated/openade-contracts.ts`, OpenADEClient product method constants, runtime-backed web product/remote method and notification references, Electron companion product/remote permission references/emits, Core product `openade/*` registrations, Core remote device/session registrations, Core product notification registrations/emits, concrete Core product/remote permission grants, Core paired profile vs Electron companion paired profile, Core idempotent mutation scopes, module product registrations/idempotency scopes/notification registrations/emits, literal Core/runtime/module JSON-RPC error codes, and referenced DTO type exports cannot drift. `check:contracts` also runs the generator's structural schema self-test and validates the real contract file shape before rendering, so unknown keys, wrong method prefixes, non-string import lists, malformed notification/error-code entries, and explicit `any` in contract type expressions fail the normal package gate. Generated Core contracts now include top-level request shape plus required/optional field type and string-literal-union validators inferred from local request literals and exported request interfaces; keep request DTOs concrete and update the contract before changing a product method boundary. `check:contracts` rejects Core product `openade/*` methods that are missing from this source, so add the contract entry before registering new product API. Product contract request/response mappings must declare every imported OpenADE DTO in their `imports` list; remote device/session mappings must use existing shared companion DTOs instead of duplicating them here. Local generated helper types belong in `localTypes`. OpenADEClient product calls, Core product registrations, module registrations, runtime-backed web `components`/`pages`/`kernel`/`remote`/`shell`/`store` capability checks, and remote device/session call sites should import generated method constants instead of copying method strings, and runtime product notification routing/emission should import generated notification constants instead of copying product notification strings.
- `src/generated/openade-contracts.test.ts` is intentionally an integration test: it launches the real Go Core, reads live runtime capabilities, and calls OpenADEClient over WebSocket. Keep contract verification on this real path rather than replacing it with mocked runtime clients.
- Product notification registrations are a closed generated set for Core and the TypeScript module. Raw runtime/process/PTY notifications may stay outside `openade-contracts.json`, but any `openade/*` product notification or `remote/device/changed` must be added there before Core/module registration or client routing.
- TypeScript OpenADE module method registrations must be a subset of generated contract methods. Core is the closed production method surface; the module may omit trusted Core-only maintenance/import routes, but it must not introduce uncontracted `openade/*` methods.
- `RuntimeClientLike` intentionally has a generated OpenADE method overload plus a raw runtime fallback overload. OpenADE product calls inside `OpenADEClient` must use the generated overload so request params and response values stay tied to `OpenADERequestByMethod` / `OpenADEResponseByMethod`; raw runtime methods such as `runtime/list` may keep the fallback overload.
- `RuntimeClientLike` implementations must expose `connect()`, current `capabilities`, and generic runtime `hasMethod()`. The public `OpenADEClient.hasMethod()` helper accepts generated `OpenADEMethod` values only. `OpenADEClient` must connect once to initialize capabilities when needed and then fail locally before sending a typed OpenADE request whose `OPENADE_METHOD` is not advertised. Keep this as a shared desktop/web/mobile backstop for stale handlers; UI should still hide denied controls from capabilities before users can trigger them. If a raw runtime helper is intentionally exposed here, such as `listRuntimes()` for runtime lifecycle hydration, route it through the same sanitized request-id and slow/burst telemetry path without adding OpenADE DTO logic to runtime-client.
- `OpenADEClient.subscribeToChanges()` forwards generated OpenADE notification methods plus runtime lifecycle and `connection/lagged` notifications needed by product-store state. Do not reintroduce broad `openade/*` or `remote/*` prefix forwarding; add the notification to `openade-contracts.json` first.
- Every typed OpenADE mutation should preserve a caller-provided `clientRequestId` or attach one by default before calling the server. Product-level retry/double-submit idempotency belongs in OpenADE, not in the low-level runtime protocol.
- OpenADEClient coalesces identical in-flight typed read requests such as task reads, project file searches, git summaries, process lists, cron definitions, and cron install-state reads. Keep that read-only; do not coalesce mutations or requests carrying `clientRequestId`.
- OpenADEClient intentionally logs slow typed and explicitly exposed raw runtime requests plus per-method request bursts to the browser/renderer console without request params or content. Burst logs count actual outbound runtime requests after in-flight read coalescing, not coalesced callers. Keep those logs to method, sanitized local request ids, client-observed timing/count, and client metadata so desktop, web, and mobile can spot product-store slamming without leaking repo paths, prompts, or task data. When the runtime transport supports caller-provided request ids, OpenADEClient must pass the same sanitized local request id into the JSON-RPC request so client slow logs can be joined with server/runtime slow logs for queue-wait versus handler-time attribution. Do not add `queueWaitMs` or `handlerMs` to the client-side log unless it comes from an authoritative server/runtime timing event.
