# OpenADE Client

Target package for typed OpenADE project/task/turn client APIs.

- Should call OpenADE runtime methods through runtime-client.
- Dashboard and mobile should converge on this package.
- Do not duplicate companion-only domain request types here.
- Runtime transport clients own `initialize`; OpenADEClient assumes it is given a runtime-client-compatible transport and only owns typed snapshot/task reads, turn-start helpers, task create/update/delete product mutations, queued-turn enqueue/reorder/cancel plus trusted legacy-import mutations, action-event and snapshot-event persistence helpers, scoped project file tree/read/write/search/process helpers, scoped project git info/branches/summary reads, scoped task terminal helpers, scoped task git scope/read/commit helpers, scoped task environment prepare, scoped task title generation, scoped task image write/read plus trusted staged-image preview reads, single/bulk legacy-import, and staged-image GC helpers, trusted MCP server settings helpers, trusted cron install-state helpers, resource-inventory/snapshot patch read plus trusted legacy-import helpers, trusted harness-session and resource import orchestration helpers, and OpenADE-scoped notification filtering.
- `openade-contracts.json` is the package's current generated contract source for OpenADEClient method names plus request/response DTO mapping. Run `npm run generate:contracts` after changing OpenADEClient methods or DTO mappings, and keep `npm run check:contracts` in typecheck so the generated `src/generated/openade-contracts.ts`, public client method strings, and Core product registrations cannot drift.
- `src/generated/openade-contracts.test.ts` is intentionally an integration test: it launches the real Go Core, reads live runtime capabilities, and calls OpenADEClient over WebSocket. Keep contract verification on this real path rather than replacing it with mocked runtime clients.
- Every typed OpenADE mutation should preserve a caller-provided `clientRequestId` or attach one by default before calling the server. Product-level retry/double-submit idempotency belongs in OpenADE, not in the low-level runtime protocol.
- OpenADEClient coalesces identical in-flight typed read requests such as task reads, project file searches, git summaries, process lists, and cron install-state reads. Keep that read-only; do not coalesce mutations or requests carrying `clientRequestId`.
- OpenADEClient intentionally logs slow typed runtime requests and per-method request bursts to the browser/renderer console without request params or content. Burst logs count actual outbound runtime requests after in-flight read coalescing, not coalesced callers. Keep those logs method/duration/client metadata only so desktop, web, and mobile can spot product-store slamming without leaking repo paths, prompts, or task data.
