# OpenADE Module

Target package for OpenADE product semantics loaded into a runtime server.

- Own projects, tasks, turns, comments, snapshots, openade.toml process config DTOs/parser/serializer, and Yjs compatibility.
- Own OpenADE product and openade.toml config types in src/types.ts; companion shared types and host bridges should only re-export or alias them for compatibility.
- `src/types.ts` owns shared queued-turn enqueue/reorder/cancel and trusted legacy-import DTOs used by core, clients, and compatibility hosts. Keep those DTOs concrete; do not route queue writes through task metadata shapes.
- `src/clientRequestIds.ts` owns stable id derivation from `clientRequestId` for OpenADE tasks and queued turns. Electron companion runtime and headless Node runtime should import these helpers instead of maintaining host-specific request-id hashing.
- Product helpers exported through `src/index.ts` must stay browser-safe. Use `src/sha256.ts` for synchronous stable hashes instead of importing `node:crypto` into helpers consumed by browser or renderer tests.
- `src/procs.ts` owns the shared dependency-light `openade.toml` parser/serializer used by Electron procs editing and Node project-process discovery. Do not add a second host-specific parser in Electron, web, or node adapters.
- `src/scopedProjectHost.ts` owns the shared Node filesystem implementation for scoped OpenADE project file tree/read/write/filename-search/content-search methods, optional task scoping, task workdir resolution, path containment, file metadata, and product fuzzy-search directory `treeMatch` results. It should prefer `rg --files` for scoped path listing, cache/in-flight-coalesce scoped path walks briefly for fuzzy/content search, and invalidate that cache after scoped file writes; do not reintroduce one filesystem walk per fuzzy-search request. Electron companion runtime and headless Node runtime should import these helpers instead of maintaining host-specific copies.
- `src/scopedProjectProcesses.ts` owns scoped OpenADE project process definition building, cwd containment from parsed `openade.toml` configs, process registration/scope matching, timeout policy, and runtime process DTO normalization. `openade/project/process/list` should also return the parsed config groups so shared cron/process UI can consume one product config read. Hosts may keep their own process lifecycle plumbing, but they should not duplicate config-to-definition, timeout, or DTO-shaping logic.
- `OpenADECronInstallState*` DTOs in `src/types.ts` are the shared per-repo cron install-state contract. Runtime/Core implementations should store them through product APIs such as `openade/cron/installState/read` and `openade/cron/installState/replace`, not through renderer `data/cron/*.json` reads. Keep these routes out of paired-device grants until Core owns scheduling permissions explicitly.
- `src/scopedTaskTerminal.ts` owns scoped OpenADE task terminal id derivation and raw PTY base64/plain-text conversion helpers. Electron companion runtime and headless Node runtime should import these helpers instead of maintaining host-specific terminal hash or encoding logic.
- `openade/task/terminal/reconnect` may omit `terminalId`; host adapters must derive the scoped terminal id from `repoId` and `taskId` server-side, then validate any provided id with `assertOpenADETaskTerminalId()`.
- `src/snapshotPatchIndex.ts` owns OpenADE snapshot patch indexing and byte slicing. Electron may re-export compatibility names, but should not maintain a second index builder.
- `src/taskSnapshotPatchReads.ts` owns inline-vs-external task snapshot patch, index, and slice read semantics. Host adapters should provide storage callbacks rather than duplicating patch-file id validation or inline/external read rules.
- `src/taskResourceInventory.ts` owns task-owned snapshot/image/session/worktree inventory DTO extraction for deep delete and resource review. Snapshot inventory must treat `patchFileId` as the resource id and fall back to the snapshot event id for legacy inline patch events that predate external patch blobs. Host adapters should provide only host-specific branch-merged status instead of duplicating task event scans.
- `openade/task/image/write` is trusted-local pre-task image staging. Hosts/Core may accept base64 image bytes and store them as product blob metadata, but `openade/task/image/read` must still prove the image id/ext is referenced by the requested task before returning bytes. `openade/task/image/staged/read` is the trusted-local pending-composer preview read for unreferenced staged blobs and must stay out of paired-device grants until scoped upload/read permissions are designed. `openade/task/image/importLegacy` and `openade/task/images/importLegacy` are trusted-local migration primitives for copying existing local image files into Core-owned `task_image` blob storage. `openade/task/images/gcStaged` is trusted-local staged-image garbage collection for unreferenced old `task_image` blobs. Do not grant these image write/import/staged-read/GC routes to paired devices until explicit upload and maintenance permissions are designed.
- `openade/snapshot/create` is trusted-local runtime plumbing. It should keep the existing snapshot event DTO shape while moving durable patch bytes into host/Core-owned blob storage; paired clients must not receive direct snapshot-create grants by default. `openade/task/snapshots/importLegacy` is a trusted-local migration primitive for copying existing local patch files into Core-owned `snapshot_patch` blob storage; do not grant it to paired devices because it accepts a raw host directory.
- `openade/task/sessions/importLegacy` is a trusted-local migration primitive for copying task-referenced Claude Code/Codex transcript JSONL files into Core-owned `harness_session` blob storage. `openade/import/legacyResources` is trusted-local migration orchestration over the image, snapshot, and harness-session import primitives. The TypeScript OpenADE module registers `openade/import/legacyResources` only when a host adapter provides `importLegacyResources`; do not add a fake no-op importer for legacy Yjs hosts. These routes may accept legacy data directories, Claude config directories, or Codex home directories, so keep them out of paired-device grants and route callers through the typed DTOs in `src/types.ts`.
- `openade/settings/mcpServers/read`, `openade/settings/mcpServers/replace`, `openade/settings/mcpServers/upsert`, and `openade/settings/mcpServers/delete` are trusted-local settings methods for MCP server rows. Core stores them in SQLite; the TypeScript OpenADE module registers them when a host adapter provides MCP settings methods. The DTOs in `src/types.ts` preserve the existing desktop MCP row shape so settings UI and migration/import code can move off direct Yjs `code:mcp_servers` access without introducing companion-specific aliases. Keep these routes out of paired-device grants because they may expose OAuth tokens and stdio commands.
- `openade/settings/personal/read` and `openade/settings/personal/replace` are trusted-local settings methods for classic desktop personal settings. Core stores them in SQLite; the TypeScript OpenADE module registers them when a host adapter provides personal-settings methods, and Electron compatibility hosts may back them with legacy `code:personal_settings` through `src/yjsMutation.ts`. Keep the DTOs in `src/types.ts` concrete, preserve explicit `false` booleans, and keep these routes out of paired-device grants because they may expose env values and device preferences.
- `src/taskEnvironment.ts` owns shared task environment setup-output DTO formatting. Host adapters may create head/worktree environments, but should not duplicate setup event text generation.
- `src/taskTitle.ts` owns browser-safe title prompt/schema/output extraction and fallback title behavior for `openade/task/title/generate`. Host adapters should call it instead of duplicating renderer title prompt or parser logic.
- Register OpenADE methods under `openade/*` on the single runtime server.
- Treat `openade/action/*` as trusted-local mutation methods unless remote permissions are intentionally narrowed and reviewed.
- Do not start a second server.
- Do not put OpenADE concepts into the low-level runtime protocol.
- `src/yjsProjection.ts` reads existing Yjs repo/task documents through a storage adapter and must normalize sparse legacy records into valid OpenADE DTOs at the boundary. Do not make renderer, Electron host, or mobile callers compensate for invalid legacy task shapes. When a storage adapter exposes `readDocumentUpdate()`, `openade/task/read` must load the task document once and project meta/events/comments/device environments from that single decoded Yjs document; do not regress to one document load per collection, and do not read `code:repos` or `code:personal_settings` on the normal task-detail path. Repo preview reads are only for legacy fallback when the task document is missing or too sparse to supply required display fields. `openade/task/read` with `hydrateSessionEvents: false` must keep task/event metadata but bound action and HyperPlan stream arrays before they cross the runtime response boundary, setting `omittedEventCount` so clients can explicitly request full hydration when users expand older history.
- `src/yjsImport.ts` owns the trusted legacy Yjs-to-product-API importer. It must read legacy data only through `OpenADEYjsProjection`, write through typed OpenADE product methods with stable `clientRequestId`s, and return explicit skipped/error items instead of silently dropping data. Keep it pointed at runtime/Core writer APIs, not renderer Yjs stores. Do not replay legacy queued turns through `openade/queued-turn/enqueue` during import because Core drains that queue and may start executions; use `openade/queued-turn/importLegacy` when the writer exposes it, and explicitly skip queued turns only for writers that have not implemented the non-executing import route. Legacy task `enabledMcpServerIds` should be preserved through task creation, and `sessionIds` plus `cancelledPlanEventId` should be preserved through `openade/task/metadata/update`; do not reintroduce skip reporting for those metadata fields.
- `npm run import:yjs:core -- --core-url ws://host:port/v1/runtime --token TOKEN --data-dir /path/to/copied/yjs` is the copied-data import/parity CLI for a running Core. It bundles `src/yjsImportCoreCli.ts` with the repo's existing Electron esbuild/yjs install, imports through real `OpenADEClient`, runs `compareOpenADELegacyYjsToCore`, prints JSON, and exits nonzero on import errors/skips, parity mismatches, or requested resource import skips/issues. Pass `--import-resources` to also run Core's trusted legacy resource importer for referenced images and snapshot patches; add `--import-sessions` plus `--claude-config-dir`/`--codex-home` when verifying copied Claude Code/Codex transcripts. When `--data-dir` is a `yjs` directory, resources default to its parent data root, otherwise use `--resources-dir` for explicit copied-data layouts. Keep the real CLI/Core test covering the canonical copied `data/yjs` layout with sibling `data/images` and `data/snapshots`. Keep it usable for read-only copied production data verification, including reruns after Core restarts; existing resource blobs should report as already imported, not conflicts or duplicates.
- `src/yjsImportParity.ts` owns reusable old-Yjs-vs-Core DTO parity reporting for the migration path. It should compare stable product semantics, including preview usage stats, task resource inventories for snapshots, images, harness sessions, and worktree identity, and normalize benign wire differences such as ISO timestamp precision, absent optional fields, and externalized patch blobs. Do not compare dynamic runtime-only inventory fields such as `isRunning` or git-derived `branchMerged` as copied-data parity. Use it with real `OpenADEYjsProjection` plus real `OpenADEClient`/Core data; do not turn it into a mocked writer assertion.
- `src/yjsMutation.ts` owns task creation, task environment setup, action-event transaction shapes, and legacy settings row mutation helpers for existing Yjs documents. Viewed-only metadata updates (`lastViewedAt`) should stay on the repo-preview fast path and must not load or save multi-megabyte task documents during route-open churn. MCP compatibility helpers should keep preserving the full desktop row shape, including stdio `cwd`, while callers migrate to the typed `openade/settings/mcpServers/*` methods. Personal-settings compatibility helpers should only be used by legacy hosts; clean Core renderer startup should use `openade/settings/personal/*` and avoid opening `code:personal_settings`.
- `src/promptBuilder.ts` owns portable prompt construction for server-owned plan/do/ask turns; host adapters may enrich the result with persisted image content blocks before harness execution.
- `src/hyperplan.ts` owns portable OpenADE-layer HyperPlan strategy validation, step ordering, prompt construction, and output extraction. Keep it out of the low-level runtime packages.
- `src/node.ts` owns the optional Node host adapter for loading OpenADE task/turn semantics into a RuntimeServer. It may use runtime-node adapters, but runtime-node must not import OpenADE product semantics.
- `src/kernel.ts` owns the reusable Node OpenADE kernel composition: RuntimeServer, runtime-node host modules, runtime-node agent module, OpenADE module, checkpoint store, and HTTP/WebSocket serving. Future agents changing shared shell, companion, desktop runtime composition, or headless serving must check this file before adding Electron-only composition.
- `src/nodeYjsStorage.ts` owns file-backed Yjs document storage compatible with existing OpenADE document ids for headless Node OpenADE hosts. Keep its legacy nested-path recovery and mismatched task-document read tolerance aligned with desktop storage so old user data remains recoverable.
- `OpenADEScopedHostAdapter` is the product-level host boundary for shared-shell host reads and tightly scoped writes. Add scoped methods here, such as `openade/project/files/tree`, `openade/project/files/fuzzySearch`, `openade/project/file/read`, `openade/project/file/write`, `openade/project/search`, `openade/project/git/*`, `openade/project/process/*`, `openade/task/terminal/*`, `openade/task/changes/read`, `openade/task/diff/read`, `openade/task/git/*`, `openade/task/image/read`, `openade/task/resourceInventory/read`, `openade/task/title/generate`, `openade/task/environment/prepare`, and `openade/task/snapshot/*` reads, instead of granting remote clients raw `fs/*`, `git/*`, `pty/*`, `process/*`, `snapshot/*`, `host/*`, or `data/*` methods. Project file/search/git requests may carry only repo ids and scoped options; hosts must resolve repo roots server-side. Project file/search requests may carry `taskId`; hosts must resolve task cwd server-side and keep paths scoped-root-relative. File writes must stay repo-relative/task-root-relative and permissioned separately from read methods. Scoped process methods must resolve `openade.toml` process ids and cwd server-side. Scoped terminal methods must derive PTY ids server-side from repo/task ids. Scoped git scope discovery must return sanitized branch/worktree metadata only; if a later git read accepts a scope id, the host must resolve that id server-side and never trust renderer-supplied cwd. Scoped title generation must resolve repo/task ids server-side, run through the host harness in read-only mode, and persist through OpenADE task metadata. Scoped environment prepare must resolve repo/task ids server-side, create/reuse worktrees from trusted host state, and persist setup through OpenADE writers. Scoped git commits, title generation, and environment prepare must remain ungranted to paired devices unless an explicit role/permission decision is made. Scoped image reads must verify the image is referenced by the task before loading bytes.
- Keep server-side writes idempotent where clients provide stable request ids or task ids.
- New head/worktree `openade/turn/start` calls should create the task document, task environment, setup event when needed, and action event before starting runtime-owned execution.
- Desktop new-task creation and programmatic OpenADE turn/Cron starts should call `openade/turn/start` through the local runtime client instead of creating task docs directly in the renderer.
- Desktop task-thread Do/Plan/Ask/Revise Plan/Run Plan actions should call the runtime-backed OpenADE turn existing-task path. Review should call `openade/review/start` so the runtime host creates the review event and follow-up Ask.
- Action event creation, stream append, execution metadata updates, complete, error, and stopped writes have server-side primitives and should be preferred over renderer-only task mutation when wiring new runtime execution paths.
- Snapshot event persistence has a server-side primitive, but patch generation and external patch-file storage are still host adapter responsibilities.
- Comment create/edit/delete has server-side primitives.
- Common task metadata updates have a server-side primitive.
- Head-mode and worktree plan/do/ask turns should persist harness session, stream, terminal status, and git ref updates through the Yjs writer.
- Headless Node plan/do/ask, HyperPlan, and review/follow-up turns should use the host-provided runtime agent executor and the same Yjs writer path; do not create alternate task/action persistence just for CLI serve mode.
- OpenADE-owned `runtime/stop` must interrupt the active host execution and persist the current action event as `stopped` through the Yjs writer before the generic runtime supervisor is allowed to mark the runtime stopped. Do not rely on a later harness abort settlement for durable task state.
- Server-owned OpenADE runtime records should include `scope.labels.eventId` and `scope.labels.executionId` for the action event they own. The generic runtime only owns lifecycle; OpenADE uses those labels to reconcile durable task history.
- Terminal runtime reconciliation belongs in this package. `openade/action/reconcileRuntime` should settle only a matching in-progress action event; missing events are not guessed, and already-terminal events are not rewritten.
- HyperPlan turns should persist the main action event plus child sub-execution lifecycle through `openade/hyperplan/*` mutation primitives.
- HyperPlan sub-execution status includes `stopped`; do not coerce an aborted child into `error` because that loses the lifecycle reason.
- Electron main now provides the host adapter path for snapshot patch generation and external patch-file storage after completed server-owned plan/do/ask turns.
- Electron main now provides host adapter paths for image prompt assembly and enabled MCP server config assembly before runtime-owned harness starts.
- Repo create/update/delete and task delete have server-side primitives. Task document deletion should remove the repo preview first and delete the task document only after host cleanup has been attempted by the embedding host.
