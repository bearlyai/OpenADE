# Web Remote UI

Renderer-side remote control surface shared by the desktop-hosted mobile view and the Capacitor companion shell.

## Architecture

- Keep only explicit legacy compatibility logic in the existing renderer/Yjs store. New remote product behavior should target the selected OpenADE runtime/Core product API through `OpenADEProductStore`.
- The desktop renderer should listen to runtime OpenADE notifications and refresh runtime product DTO caches after server-owned writes. Do not add new remote paths that refresh cached Yjs documents for normal product behavior.
- Remote invalidations come from runtime OpenADE notifications; do not reintroduce renderer-origin companion event bridges. Runtime notification routers and refresh policies should import `OPENADE_NOTIFICATION` from `@openade/openade-client` instead of copying product notification strings.
- Remote task-detail invalidations include `openade/task/updated` and `openade/queuedTurn/updated` only when they target the currently selected task; unrelated task and queued-turn updates must not refresh the visible task. Notification-driven task and snapshot refreshes must bypass `OpenADEProductStore` completed-result caches so remote/mobile views do not keep showing stale DTOs after Core or desktop-side writes.
- Accepted remote mutations that carry a `clientRequestId` may suppress matching echo notifications with the same `clientRequestId` after `OpenADEProductStore` patches the local cache. Do not suppress untagged task notifications or tagged notifications from other request ids; those are external state changes and must still refresh through the runtime path.
- `RemoteApp.tsx` is the remote session adapter and should stay thin over `client.ts`, shared session helpers, and shared shell components.
- `projects/web/src/kernel/session.ts` owns shared pairing URL parsing, private-host validation, runtime WebSocket URL construction, runtime client caching, and OpenADE client construction. `client.ts` owns companion config persistence, `getRemoteProductStore()`, `retryRemoteRead()`, and remote-only device/session actions on top of that shared session layer.
- Remote/shared shell visibility and stale-handler guards should derive advertised methods through `getRemoteRuntimeCapabilities()`. Do not add ad hoc runtime-method helpers or medium-specific capability checks in `RemoteApp.tsx`.
- Scoped project/task/new-task handlers in `RemoteApp.tsx` must also re-check the active config plus current selected repo/task/new-task refs at action time before issuing runtime requests; task-scoped handlers and delayed task mutation results must also require the matching shell screen to still be active. Do not rely only on render-closure ids after navigation or session switches.
- Recompute the shell capability projection from the selected runtime entry during render rather than memoizing it across capability initialization. Some local/runtime clients can finish `initialize` without a distinct visible status transition; the next DTO render must see the initialized method set so snapshots/project lists are not hidden behind stale false capabilities.
- Remote/session actions that call generated `OPENADE_REMOTE_METHOD` values must go through `requestKernelRemoteMethod()`. The helper connects to load runtime capabilities when needed and fails locally before sending the request if the method is not advertised, so stale session-shell handlers cannot create hidden denied remote/admin requests.
- `projects/web/src/kernel/productStore.ts` owns runtime-backed product DTO caching and product mutations. RemoteApp product controls should call `OpenADEProductStore` through `getRemoteProductStore()` instead of bespoke companion commands or per-method `readRemote*`/`startRemote*` aliases.
- OpenADE product types should be imported directly from `projects/openade-module/src`; `projects/shared/companion` is only for companion-owned pairing/device/service DTOs.
- client.ts must cache one runtime WebSocket client per paired host id; navigation, refresh, and multiple subscriptions must not create extra sockets for the same saved credentials.
- Shared `projects/web/src/shell/task/taskEventPresentation.ts` turns task events into message and activity rows; keep remote-specific code limited to image loading/upload persistence and session wiring.
- Task-thread scroll behavior is shared with desktop through `projects/web/src/shell/task/useTaskThreadScroll.ts`; do not add remote-only bottom-follow logic back into `RemoteApp.tsx`.
- Task command labels, mode ordering, queueability rules, and desktop command-state descriptors are shared through `projects/web/src/shell/task/taskCommands.ts` and `projects/web/src/shell/task/taskCommandModel.ts`; do not reintroduce remote-only command label or enablement helpers.
- Task input should use the desktop-derived `SmartEditor` through shared shell composer slots, not a remote-only textarea. RemoteApp may supply the editor slot with capability-scoped product file search and SDK slash-command managers; keep those reads lazy from actual editor interactions.
- Remote composer/editor wiring must stay browser-safe. `RemoteApp` should instantiate `remote/remoteSmartEditorManagers.ts` for file mentions and SDK slash commands instead of importing `store/managers/SmartEditorManager`, `store/managers/SdkCapabilitiesManager`, or `electronAPI/*`. `SmartEditor` should depend on the exported editor/capability contracts, with desktop managers remaining structurally compatible.
- Remote route composition lives in `projects/web/src/shell/OpenADEShell.tsx`; `RemoteApp.tsx` should own pairing/session state, runtime refresh, and action handlers, then pass DTOs and callbacks into the shared shell wrapper. Do not add a separate mobile product UI layer; mobile is only a Capacitor/native host adapter around this shell.
- The browser `/remote` entrypoint must render without eagerly importing the desktop route/store graph, constructing or mounting the desktop `CodeStore`, importing `electronAPI/*`, loading raw PTY/data-folder/capability helpers, or probing local Electron/runtime IPC. Keep `projects/web/src/App.tsx` thin over `RemoteApp` plus a lazy desktop `CodeApp`, and keep `projects/web/src/main.tsx` as a mount-only entrypoint. Only desktop code routes should load `CodeApp`, create a `CodeStore`, enter the `CodeStoreProvider`, or hit desktop-only Electron helpers; remote/mobile companion boot should stay a thin runtime-attached shell path.
- Browser-safe shared components may keep desktop fallback behavior only behind lazy imports guarded by `VITE_OPENADE_ENABLE_DESKTOP_FALLBACK_CHUNKS`. Mobile sets that flag to `false`, so remote-compatible components must use runtime product adapters for terminals/images and return fail-closed instead of importing raw Electron PTY, Electron shell, or legacy data-folder modules.
- Project task lists, project file/search/git/cron/process panels, new-task form, full task route composition, task changes/resource panels, task event rendering, task composer controls, and task metadata/review/comment/queued-turn controls live under `projects/web/src/shell/*`; keep new reusable product panels there instead of adding more product-control JSX directly to `RemoteApp.tsx`.

## Data Flow

- Electron main owns companion auth and runtime transport in legacy desktop-hosted sessions. When Core is configured, Electron's companion listener should only bridge pairing/runtime sockets to Core, and desktop companion settings should use generated remote methods through the selected product runtime. OpenADE Core is the target owner for product state, host operations, permissions, and notification filtering; do not add new companion-only product projection logic in Electron main.
- The desktop renderer is not a companion command adapter. Product commands from remote clients go through runtime WebSocket methods.
- Remote clients read snapshots, projects, task detail, and runtime updates over the `/v1/runtime` WebSocket. Initial attach may try `openade/snapshot/read` through `OpenADEProductStore` so the runtime initializes and desktop theme/server context is loaded, but if that method is not advertised it must fall back to `openade/project/list` without issuing a denied snapshot request.
- Remote product mutations include capability-gated project create/update/archive/delete, task create, turn start/interrupt, review start, queued-turn enqueue/cancel/reorder, comment create/edit/delete, task metadata update, and capability-gated plain task delete. Keep them high-level OpenADE methods; never replace them with raw Yjs or host calls. Shared project management may call `openade/repo/create`, `openade/repo/update`, and `openade/repo/delete` only when those capabilities are advertised; default paired clients should keep them hidden because repo path management is host authority. The shared New Task route must use `openade/task/create` for the task record, then optionally attach an execution through `openade/turn/start` only when that capability is granted. The running-task composer must use `openade/queued-turn/enqueue` for queueable Do/Ask/HyperPlan input instead of relying on turn-start fallback behavior. Remote task delete must not expose cleanup flags unless a future runtime capability explicitly grants that host-cleanup permission.
- Accepted task mutations may update runtime state after the user navigates away. Post-mutation task refreshes for metadata, comments, queued turns, title generation, and environment preparation must require the same active task shell scope before reading or repainting task DTOs.
- Shared composer MCP selection may read `openade/settings/mcpServers/read` only when that capability is advertised. Default paired clients may receive that read grant because Core and the TypeScript OpenADE module redact read-only responses to connector summaries; the shared shell must still avoid rendering URLs, commands, args, cwd, headers, OAuth tokens, or env vars unless write capabilities are explicitly granted. `RemoteApp` may provide MCP picker slots, but `OpenADEShell` owns visibility from `openade/settings/mcpServers/read` plus task metadata/create capabilities and must strip stale MCP slots when those capabilities disappear. New task selection should flow through `openade/task/create` and optional `openade/turn/start`; existing task selection should flow through `openade/task/metadata/update`.
- Shared Settings may render product preference and connector summaries through `openade/settings/personal/read` and `openade/settings/mcpServers/read` only when those capabilities are advertised. Keep personal-settings reads lazy to Settings and New Task default-agent initialization, keep connector reads lazy to Settings or explicit composer use, and show counts/statuses instead of env var values, headers, OAuth tokens, URLs, or commands by default. Personal-settings and connector loaders must re-check the active config and capability at call time before issuing the runtime request, so stale Settings/New Task effects cannot read from a disconnected or newly denied session. Personal env vars may be revealed and edited only after an explicit user action and only when `openade/settings/personal/replace` is advertised; save the full typed `OpenADEPersonalSettings` replacement through the runtime product API and keep read-only profiles on count-only rendering. Connector enable/disable/create/edit controls may call `openade/settings/mcpServers/upsert` only when that capability is advertised, and connector delete may call `openade/settings/mcpServers/delete` only when that capability is advertised. The shared connector form may edit name, enabled state, transport on create, HTTP URL, stdio command, stdio args, stdio cwd, and explicit Advanced JSON values for HTTP headers or stdio env vars only under the upsert capability. Do not render those Advanced secret editors in read-only profiles. Do not render or edit connector OAuth token fields in the shared shell; preserve existing OAuth tokens when editing HTTP connector metadata. Broader settings-admin controls need explicit write capabilities plus real runtime coverage.
- Shared new-task worktree selection may read `openade/project/git/branches/read` only when that scoped capability is advertised and the user explicitly loads branches. Do not load branches on new-task screen open. When selected, send `{ type: "worktree", sourceBranch }` through `openade/task/create`; otherwise send `{ type: "head" }`.
- Shared Run Plan and Revise Plan are active-plan commands. Show and submit them only when the hydrated task has an active completed plan/revise/HyperPlan event that has not been run, followed by Do, or cancelled.
- Shared HyperPlan strategy presets are shell state that must become a concrete `hyperplanStrategy` on `openade/turn/start` for existing tasks/create-and-run tasks and on `openade/queued-turn/enqueue` when the selected task is already running. Do not start or queue a HyperPlan turn without a valid serialized strategy, and do not move HyperPlan execution ownership back into the renderer.
- Shared Retry is a classic task command, not a separate runtime method. `OpenADEShell` derives retry visibility from the hydrated task events, current running state, and `openade/turn/start`; do not pass separate medium-owned retry booleans into the shell. Submit `ACTION_PROMPTS.retry` as a `"do"` turn with `label: "Retry"` and `includeComments: false`, and keep the remote handler's action-time guard on the same shared retry predicate.
- Shared Cancel Plan is task metadata, not a separate runtime method. Show it only when there is an active completed plan/revise/HyperPlan event and `openade/task/metadata/update` is advertised; write that event id as `cancelledPlanEventId`.
- Shared task/new-task image attachment controls may appear only when `openade/task/image/write` is advertised and the submitted action can actually start or queue. Use `processImageBlob()` plus `OpenADEProductStore.writeTaskImage()` to persist resized bytes first, then include the returned image references on `openade/turn/start` or `openade/queued-turn/enqueue`. Default paired clients must keep upload hidden unless that exact scoped write capability is granted; `openade/task/image/read` by itself is only enough for rendering referenced images. Do not pass separate medium-owned `canAttachImages` booleans into `OpenADEShell`; the shell derives upload visibility from generated runtime capabilities plus available handler slots, and handlers must still re-check capabilities at action time.
- Delayed task/new-task image processing results must re-check the same active shell scope before adding attachment previews. If the user leaves the task or New Task screen while `openade/task/image/write` is in flight, drop the processed preview and revoke its object URL instead of letting stale attachments appear when the user returns.
- Shared task/new-task `SmartEditor` file mentions may call `openade/project/files/fuzzySearch` only when that capability is advertised, and slash-command discovery may call `openade/project/sdkCapabilities/read` only when that product capability is advertised. Do not issue fuzzy-search or SDK-capability reads from route open, task switch, or composer mount.
- Remote project files and search use scoped `openade/project/files/tree`, `openade/project/files/fuzzySearch`, `openade/project/file/read`, and `openade/project/search` methods through `OpenADEProductStore`; do not call raw `fs/*`, `host/*`, or direct repo paths from `RemoteApp`. Keep file-tree loading and file-name fuzzy search lazy/manual from the Files panel so project opens stay on lightweight task-list DTO reads. A fuzzy-search result may read a file without first loading the whole file tree. Accepted project-file writes should patch the open file from the write result; do not refresh the file tree unless the user explicitly loads or refreshes the Files panel.
- Remote/shared shell panels and sub-actions must derive visibility and handlers from runtime `initialize.capabilities`. Missing base read capabilities hide the panel; missing sub-action capabilities hide or disable only that action and handlers must return before issuing a denied runtime request.
- Remote project git views use scoped `openade/project/git/info/read`, `openade/project/git/branches/read`, and `openade/project/git/summary/read` through `OpenADEProductStore`; keep this lazy/manual from the project git panel so project opens do not pay git status cost. Gate info, branches, and summary independently from `initialize.capabilities`; the shell may show a useful partial Git panel, but `RemoteApp` must only call the specific methods the runtime advertised.
- Remote project cron definition views use scoped `openade/cron/definitions/read` through `OpenADEProductStore`. Cron install-state controls may read/replace through `openade/cron/installState/read` and `openade/cron/installState/replace` only when those capabilities are advertised; paired clients must keep them hidden unless a runtime explicitly grants them. Cron run-now controls may call `openade/cron/run` only when that capability is advertised; default paired/browser/mobile roles should keep it hidden unless a role explicitly grants that trusted action.
- Remote task change views use scoped `openade/task/changes/read`, `openade/task/diff/read`, `openade/task/filePair/read`, `openade/task/git/summary/read`, `openade/task/git/scopes/read`, `openade/task/git/log`, `openade/task/git/commit/files/read`, `openade/task/git/commit/filePatch/read`, and `openade/task/git/fileAtTreeish/read` through `OpenADEProductStore`; do not call raw `git/*` or `fs/*` from `RemoteApp`. Task opening should not eagerly load git data; keep those reads lazy/manual from the task git panel so normal task switches stay on the lightweight task DTO path. Gate changes, summary, scopes, and log independently from `initialize.capabilities`; partial task Git profiles may render partial data, but `RemoteApp` must only call the specific task Git methods the runtime advertised, and the shared shell must not keep rendering stale git payloads after a matching capability disappears.
- Delayed task-detail route reads, task panel reads, and task mutations for git, resources, snapshot patches, comments, metadata, queued turns, reviews, and task execution must require the same active task shell screen before applying task DTOs, panel payloads, loading flags, action markers, notices, or errors. Server-side task actions may still complete after navigation, but stale completions must not repaint hidden task panels, show stale route errors, keep the destination screen globally loading/disabled, or prefill task data for the next task visit. Remote submission locks are token-scoped and must be cancelled when the selected screen/repo/task changes so a stale `finally` block cannot clear a newer action or keep the next shell screen spinning.
- Remote snapshot event patch views use scoped `openade/task/snapshot/index/read` plus `openade/task/snapshot/patch/readSlice` through `OpenADEProductStore` when both capabilities are advertised, falling back to `openade/task/snapshot/patch/read` for older runtimes. Keep these reads lazy/manual from the snapshot block so task opening stays on lightweight task DTOs.
- Remote task resource views use scoped `openade/task/resourceInventory/read` through `OpenADEProductStore`; keep this lazy/manual from the task resource panel because inventory can touch runtime state and git metadata.
- Remote task terminal views are capability-gated and use scoped `openade/task/terminal/start`, `openade/task/terminal/reconnect`, `openade/task/terminal/write`, `openade/task/terminal/resize`, and `openade/task/terminal/stop` through `OpenADEProductStore` and the shared Terminal product adapter; do not call raw `pty/*` from `RemoteApp`. Keep terminal mounting lazy/manual so task opening does not start or reconnect PTYs. Gate terminal start, reconnect, write, resize, and stop independently from `initialize.capabilities`: reconnect-only profiles may open a read-only terminal/output view, start-only profiles may start without polling output, input/resize/stop must no-op unless those methods are advertised, and restart controls require both start and stop.
- Remote project process controls and output viewing use scoped `openade/project/process/*` methods through `OpenADEProductStore`; do not call raw `process/*` or `host/procs/*` from `RemoteApp`. Project navigation should not eagerly call `openade/project/process/list`; keep process-list loading lazy/manual from the process panel's explicit load/refresh controls so normal project opens, output viewing, and accepted start/stop results do not pay the host process-config cost. Render process start, output reconnect, and stop independently from runtime capabilities because paired clients may be read/output-only.
- Delayed project panel reads and mutations for files, search, git, cron, and process controls must require the same active project shell screen before applying panel DTOs, loading flags, action markers, notices, or errors. Server-side project actions may still complete after navigation, but stale completions must not repaint hidden project panels.
- Remote project task preview refreshes should use scoped `openade/task/list` through `OpenADEProductStore` when that capability is advertised. Coalesce notification-driven preview refreshes per repo so a burst of `openade/task/previewChanged` notifications causes one task-list read for that repo instead of one read per notification. Repo/snapshot project-projection refreshes should use scoped `openade/project/list` once an initial snapshot already exists, preserving the snapshot's server/theme and working-task ids; keep full snapshot refreshes for initial attach, explicit manual refresh, session/global repair, and runtimes without project-list/task-list support. If `openade/snapshot/read` is not advertised, synthesize the minimum shell snapshot from `openade/project/list` instead of probing the denied snapshot method. Do not use a full snapshot refresh when only the selected project's task list or the project list needs to be fresh. Deletion notifications for a selected task must cancel pending delayed task-detail refreshes for that task/repo before scheduling the task-list or snapshot repair, so a deleted task cannot still trigger a stale `openade/task/read`.
- Remote settings may call `remote/device/selfRevoke` for the current paired token only, through the generated `OPENADE_REMOTE_METHOD` constant from `projects/openade-client`. Do not add other device-management methods to the remote shell without explicit admin-role permissions and integration tests.
- Saved inactive sessions must not be snapshot-refreshed just because the remote shell mounted or the config list changed. The active session owns the initial snapshot read; inactive sessions should show an explicit Open Session action and load through the selected runtime client only after the user switches to them.
- Delayed reads from inactive or previously active sessions may update that saved session's cached snapshot, but they must not mutate the active shell state, selected project/task, loading markers, or current task/project DTOs after the user switches sessions.
- Do not sync Yjs to the phone for the companion MVP.
- Do not expose raw Electron APIs, filesystem APIs, or shell APIs to RemoteApp.

## Important Learned Fixes

- Pairing uses HTTP URLs and full-link paste. Deep-link pairing is intentionally rejected. Preserve `hostId` from pairing links through the confirm path so saved sessions keep stable host identities instead of falling back to base-url ids.
- Multiple OpenADE sessions are stored in a versioned config store with an active id. Keep inactive host reads lazy so a phone/browser with several saved desktops does not fan out snapshot requests on launch.
- Snapshot and task payloads must be plain JSON. Remote reads should use the selected OpenADE runtime/Core product API through `OpenADEProductStore`; Yjs-backed projections are legacy compatibility only.
- If a task document is missing or has mismatched metadata, return a readable unavailable state instead of crashing the shared shell.
- Task sorting must match the desktop sidebar. Use sortTaskPreviewsLikeSidebar from components/sidebar/taskSorting.
- Harness ids from remote input must be validated at the OpenADE runtime boundary before starting a host harness.
- RuntimeClient should reconnect automatically and surface online/reconnecting/offline state in the UI.
- Settings controls must not use picker components that call `onChange` from normalization effects on mount. In Settings, `onChange` means persist through `openade/settings/personal/replace`, so use controls that write only after explicit user input.

## UI And Theme

- Follow projects/web/src/_docs/design.md and projects/web/src/tw.css.
- Read tw.css before changing colors.
- The remote UI should feel like the desktop code module: flat, square, clean, spacious.
- No rounded corners. Keep any floating shadow limited to true floating controls.
- Use btn on every button and input on every input/textarea/select-like native control.
- Use semantic theme tokens only. Pair solid bg-primary, bg-error, bg-success, bg-warning, and bg-info with their *-content text color.
- Default to matching snapshot.server.theme.className from the connected desktop.
- Keep the local remote-shell theme override and a way to switch back to matching desktop.
- Avoid visible explanatory copy. The UI should expose controls directly.

## Testing

- Run npm run typecheck from projects/web after changes.
- Add Vitest coverage for parsing, persistence migration, runtime reconnect, message presentation, task sorting, and product-control changes. Prefer real `RuntimeServer`/`RuntimeLocalClient`/`OpenADEClient` paths for integration confidence.
- `App.test.ts` guards the web route boundary: `/remote` must render without constructing the desktop `CodeStore`, while desktop code routes must still lazy-load the desktop app and initialize the store. Update that regression if the web/mobile companion entrypoint changes.
- After route or shared-shell import changes, run `npm run smoke:remote-route` from `projects/web` or otherwise smoke `http://127.0.0.1:7000/#/remote` in a real browser and watch network/console output. The smoke should show the companion Connect screen with no warnings/errors and no requests for `CodeApp`, desktop route/store modules, `store/managers/*`, `electronAPI/*`, or raw PTY helpers.
- After mobile-affecting shared-shell import changes, run the mobile production build or inspect an existing build artifact for desktop-only imports. `projects/mobile/dist/assets` should not contain `electronAPI`, `dataFolder`, `rawPtyTerminalSession`, `CodeApp`, `store/managers`, `setTerminalKeyboardCapture`, or `PtyHandle`.
- `remote/RemoteApp.integration.test.ts` renders `RemoteApp` against real in-memory runtime clients and should be extended when adding remote product controls or session shell behavior. It now covers multi-session switching, stale-host removal, inactive-session lazy snapshot reads, and local shell theme persistence through separate `RuntimeServer` instances selected by the production runtime URL builder.
- Keep browser remote-shell tests browser-safe. Do not import `runtime-node` HTTP/WebSocket server helpers into `projects/web/src/**/*.test.ts`; the web test runner bundles tests for Chromium and cannot carry Node `ws` server dependencies. Pair focused shell coverage here with `projects/openade-module/src/kernel.integration.test.ts` when a real WebSocket `RuntimeClient`/`OpenADEClient` transport proof is needed.
- For visual changes, also build projects/mobile, sync iOS, and inspect the simulator.
